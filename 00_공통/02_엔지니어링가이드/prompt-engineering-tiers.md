# 프롬프트 엔지니어링 적용 티어 가이드

> **작성일:** 2026-04-19
> **용도:** 질문의 복잡도에 따라 프롬프트 기법을 어디까지 적용할지 판단하는 실전 레퍼런스
> **원칙:** Garbage In, Garbage Out — 기법의 양이 아니라 태스크에 맞는 투자량이 핵심

---

## 판단 기준: 두 가지 축

모든 티어 판단은 아래 두 축의 조합으로 결정된다:

| 축 | 낮음 | 높음 |
|----|------|------|
| **① 실패 비용** | 틀려도 다시 하면 됨 (밥 추천) | 틀리면 시간·돈·프로젝트가 날아감 |
| **② 사고 방향 설정 필요** | 정답이 하나 (사실 확인) | 관점·순서에 따라 결론이 달라짐 (분석, 설계) |

- 둘 다 낮음 → Tier 0
- 한쪽만 높음 → Tier 1~2
- 둘 다 높음 → Tier 3

---

## 🟢 Tier 0 — 맨손 (기법 투자 = 0)

### 적용 상황
일상 대화, 단순 사실 확인, 가벼운 의견 요청.
모델이 이미 잘 아는 영역이고, 출력 형식도 상관없고, 틀려도 리스크가 낮은 경우.

### 예시

```
"오늘 밥 뭐 먹을까"
"이 단어 무슨 뜻이야"
"파이썬에서 list랑 tuple 차이 뭐야"
"졸린데 커피 마셔도 될까"
```

### Tier 0.5 — 맥락만 추가

기법은 아니지만, 한두 줄의 맥락을 붙이면 답변 품질이 확 달라지는 경우.

```
"밥 뭐 먹을까" → Tier 0
"나 매운 거 싫어하고 오늘 비 오는데 밥 뭐 먹을까" → Tier 0.5
```

---

## 🟡 Tier 1 — 구조화만 (맥락 + 범위 + 형식)

### 적용 상황
답변 품질이 맥락에 따라 달라지는 질문.
정보 전달이 핵심이고, 모델의 사고 방향을 크게 설정할 필요는 없는 경우.

### 지킬 것
- **구조화:** 뭘 알고 싶은지 구체적으로 (범위 + 관점 지정)
- **맥락 제공:** 코딩이면 에러 로그 + 환경, 학술이면 현재 지식 수준
- **출력 형식 힌트:** "표로 비교해줘", "단계별로" 정도

### 예시

**유기화학 구조식 설명 요청:**

```
❌ "이 구조식 설명해줘" [이미지만 던짐]
   → 모델이 뭘 설명할지 모름. 명명법? 반응성? 입체화학?

✅ "이 분자에서 conjugation이 가능한 부분 표시하고,
    왜 이 diene이 Diels-Alder에 적합한지 설명해줘"
   → 범위(conjugation) + 관점(Diels-Alder 적합성) + 기대 출력이 명확
```

**코딩 트러블슈팅:**

```
❌ "SSH 안 돼 왜 그래"

✅ "데스크톱(Windows 11, WSL2 Ubuntu)에서 노트북(WSL Ubuntu)으로
    Tailscale SSH 접속 시 아래 에러 발생:

    Permission denied (publickey)

    시도한 것:
    - ssh-keygen으로 키 재생성
    - authorized_keys에 공개키 추가
    환경: Tailscale v1.78, OpenSSH 9.6"
```

**논문 요약:**

```
❌ "이 논문 요약해줘"

✅ "이 논문의 Methods 섹션에서 사용한 통계 기법만
    학부 수준으로 설명해줘. 특히 sample size 산정 근거가
    타당한지 판단해줘"
```

### Tier 1의 핵심
"뭘 모르는지"를 아는 상태에서 질문하는 것.
기법이라기보다 좋은 질문 습관에 가깝다.

---

## 🟠 Tier 2 — 선택적 조합 (상황에 맞게 기법 골라 쓰기)

### 적용 상황
분석, 비교, 설계처럼 여러 관점을 동시에 고려해야 하는 태스크.
"그냥 물어보면 답은 나오는데, 답의 방향과 깊이가 불만족스러운" 경우.

### 사용하는 기법 개념 정리

**CoT (Chain-of-Thought):**
모델에게 "이 순서로 생각해"라고 사고 경로를 설계해서 강제하는 것.
없으면 모델이 어떤 축을 먼저 보고 어떤 순서로 판단할지가 랜덤이다.
비유: "서울에서 부산 가" vs "경부고속도로 → 대전 경유 → 부산"의 차이.

**Self-consistency:**
같은 질문을 서로 다른 관점에서 각각 독립적으로 답하게 한 다음,
겹치는 결론을 신뢰하는 것. GPS 삼각측량과 같은 원리.
위성 하나로 위치를 잡으면 오차가 크지만,
서로 다른 위성 3개가 동의하는 지점은 신뢰도가 높다.

### 상황별 기법 선택 요약

| 상황 | 추천 기법 | 선택 이유 |
|------|----------|----------|
| A vs B 비교 | 구조화 + Self-consistency | 단일 관점 편향 방지 |
| 아키텍처 설계 | Role + CoT | 전문가 렌즈 + 사고 순서 |
| 발표자료 구조 | 구조화 + Few-shot | 출력 형식 통제가 핵심 |
| 방법론 검토 | Role + CoT + 구조화 | 학술적 관점 + 추론 경로 |

---

### 사례 1: A vs B 비교 분석 → 구조화 + Self-consistency

비교 분석에서 가장 흔한 실패는 "모델이 한쪽에 앵커링되는 것"이다.
먼저 언급된 쪽을 유리하게 쓰는 경향이 있으며,
Self-consistency로 관점을 분리하면 이를 방지할 수 있다.

```
❌ 그냥 물어보기:
"VOO랑 SCHD 중에 뭐가 나아?"
→ 모델이 자기 편향대로 하나 추천하고 끝

✅ 구조화 + Self-consistency 적용:
"VOO vs SCHD를 비교해줘.

아래 3가지 관점에서 각각 독립적으로 분석해:
관점 A — 성장성 중심: 10년 후 자산 극대화가 목표인 투자자
관점 B — 현금흐름 중심: 배당으로 월 생활비 보조가 목표인 투자자
관점 C — 리스크 중심: 최대 낙폭(MDD)을 최소화하고 싶은 투자자

각 관점에서 승자를 판정하고,
마지막에 3가지 결론을 종합해서 내 상황(학부생, 장기 DCA,
월 소액)에 맞는 권고를 줘."
```

모델이 "성장성으론 VOO, 현금흐름은 SCHD, 리스크는 VOO" 식으로
각각 독립 판정을 내리고, 최종 종합에서 근거가 다각적으로 보인다.

---

### 사례 2: 아키텍처 설계 → Role + CoT

설계는 "정답"이 없고 트레이드오프의 연속이다.
Role로 전문가 렌즈를 씌우면 현실적 제약을 반영하고,
CoT로 사고 순서를 잡으면 "요구사항 → 제약 → 설계 → 검증" 흐름이 빠지지 않는다.

```
❌ 그냥 물어보기:
"데스크톱이랑 노트북 SSH로 연결하고 싶어"
→ 일반적인 SSH 튜토리얼이 나옴. 내 환경에 안 맞을 수 있음

✅ Role + CoT 적용:
"너는 홈랩 인프라 설계 경험이 풍부한 DevOps 엔지니어야.

내 환경:
- 데스크톱: Windows 11 + WSL2 Ubuntu, RTX 5070 Ti, 64GB RAM
- 노트북: WSL Ubuntu, user: atomicpotato
- 네트워크: Tailscale로 연결됨
- 목표: 노트북에서 데스크톱 GPU 자원 활용 (Codex CLI 등)

아래 순서로 설계해줘:
Step 1: 현재 환경에서 가능한 연결 방식 2-3가지 나열
Step 2: 각 방식의 트레이드오프 (보안, 속도, 설정 복잡도)
Step 3: 내 상황에 최적인 방식 1개 선택 + 근거
Step 4: 선택한 방식의 구현 단계 (어떤 머신에서 어떤 명령)"
```

CoT의 Step 1→4가 핵심이다. 이게 없으면 모델이 바로 구현으로 뛰어넘어서
"왜 이 방식인지"를 설명하지 않는다. 설계에서 근거 없는 구현은 위험하다.

---

### 사례 3: 발표자료 구조 → 구조화 + Few-shot

발표자료는 "내용"보다 "형식"이 품질을 좌우한다.
구조화로 제약을 걸고, Few-shot으로 "이런 수준의 결과물을 원한다"를 보여주는 게 핵심이다.

```
❌ 그냥 물어보기:
"PBL 발표자료 만들어줘"
→ 모델이 자기 판단으로 슬라이드 10장짜리 개요를 뱉음

✅ 구조화 + Few-shot 적용:
"PBL 중간발표 슬라이드 구조를 잡아줘.

조건:
- 분량: 8-10슬라이드
- 발표시간: 10분
- 청중: 교수 1명 + 같은 PBL 수강생
- 주제: adzuki bean pangenome 구축 전략

아래는 다른 주제의 좋은 발표 구조 예시야:

<example>
주제: 신약 타겟 스크리닝 파이프라인
1. 문제 정의 (1장) — 왜 이 타겟인가
2. 기존 접근법 한계 (1장) — 뭐가 부족한가
3. 우리 방법론 (2장) — 어떻게 다르게 하는가
4. 예비 결과 (2장) — 뭘 확인했는가
5. 남은 과제 + 타임라인 (1장)
6. Q&A 유도 슬라이드 (1장) — 토론할 질문 제시
</example>

이 구조적 패턴(문제→한계→방법→결과→과제)을
내 주제에 맞게 변환해줘."
```

Few-shot 예시가 다른 도메인(신약)인 게 포인트다.
같은 도메인 예시를 주면 모델이 내용까지 복사하지만,
다른 도메인이면 구조 패턴만 전이된다.

---

### 사례 4: 방법론 검토 → Role + CoT + 구조화

방법론 검토는 ① 해당 방법론의 전문 지식이 필요하고(Role),
② 전제 확인 → 위반 판단 → 대안 제시의 논리 흐름이 있고(CoT),
③ 입력 정보가 복잡해서 정리가 필요하다(구조화).

```
❌ 그냥 물어보기:
"AHP 가중치 이거 맞아?"
→ "네 CR이 0.1 미만이니 괜찮습니다" 수준의 피상적 답변

✅ Role + CoT + 구조화 적용:
"너는 MCDA 방법론 전문가야. Saaty(1980) AHP 원전과
이후 확장(ANP, TOPSIS)에 정통해.

내 AHP 결과:
| Dimension | Weight |
|-----------|--------|
| Market & Revenue | 0.3629 |
| Genomic Opportunity | 0.2193 |
| Breeding & Delivery | 0.2193 |
| Strategic Fit | 0.0921 |
| Compliance | 0.1065 |
CR = 0.0131

맥락: 종자회사의 genome 투자 대상 선정용.
      6개 후보 작물을 이 weight로 평가.

아래 순서로 검토해줘:
Step 1: AHP에서 이 CR 수준이 의미하는 것
Step 2: weight 분포가 의사결정 구조로서 합리적인지
        (D2와 D3가 동일한 것이 의도적 설계인지 문제인지)
Step 3: 이 weight를 6개 작물에 동일 적용할 때
        criteria-alternative 독립성 전제가 유지되는지
Step 4: 문제가 있다면 수정안, 없다면 타당성 근거"
```

Step 2에서 "D2와 D3가 동일한 것"을 구체적으로 짚어주는 것이 핵심이다.
이게 없으면 모델은 CR만 보고 "0.1 미만이니 OK" 하고 넘어간다.
의심하는 지점을 CoT에 명시적으로 심어야 모델이 그 지점을 깊이 판다.

## 🔴 Tier 3 — 풀장비 (7가지 기법 전부 동원)

### 적용 상황
한 번의 응답으로 끝나지 않는 복잡한 프로젝트.
실패 시 리스크가 크고, 다단계 분석이 필요한 경우.

### 예시 태스크
- PBL MCDA 종자 선정 프레임워크 설계
- 멀티에이전트 파이프라인 아키텍처 설계
- 투자 분석 시스템 전체 설계
- 논문급 문헌 리뷰 + 메타합성

### 적용 기법 (전부)
- 🎯 불변 앵커 — scope drift 방지
- Role — 도메인 전문성 설정
- CoT 루프 — `Thought → Search → Observation → Analyze → Conclude`
- Few-shot — 출력 품질 기준선 설정
- Self-consistency — 삼각측량 (다중 관점 교차검증)
- Chaining — 단계 분할, 이전 출력을 다음 입력으로 명시적 참조
- ReAct / Pseudo-ReAct — 근거 추적, 정보 충분성 판단 루프

### Tier 2와의 핵심 차이
Tier 2는 한 번의 프롬프트로 끝낼 수 있다.
Tier 3은 여러 프롬프트가 서로를 참조하면서 누적되는 구조가 필요하다.
이것이 Chaining이 필수가 되는 이유다.

→ 구체적인 Tier 3 프롬프트 예시는 `prompt-engineering-guide.md` 섹션 4 참조.

## 과잉 프롬프팅의 부작용

기법을 많이 쓴다고 항상 좋은 게 아니다.

- 모델이 "이 사람이 정교하게 요청했으니 복잡한 답을 내놔야 한다"고 해석하여, 단순한 답이면 될 걸 불필요하게 부풀리는 경향이 생긴다.
- 프롬프트 작성에 드는 시간 자체가 비용이다. Tier 0 질문에 Tier 3 프롬프트를 쓰면 답을 얻기 전에 프롬프트 쓰다 지친다.
- 과도한 제약은 모델의 유연성을 죽인다. 탐색적 질문에는 오히려 열린 프롬프트가 효과적일 수 있다.

토끼 잡는 데 40mm 보포스를 가져가면, 토끼만 없어지는 게 아니라 주변 밭까지 날아간다.

## 부록: Tier 1에서 프롬프트 기법보다 중요한 것 — 컨텍스트 품질

특히 코딩 트러블슈팅에서는 기법보다 컨텍스트가 결정적이다:

1. **에러 메시지 전문** — 잘라내지 말 것
2. **관련 코드 블록** — 전체 파일이 아니라 해당 부분
3. **환경 정보** — OS, 런타임 버전, 패키지 버전
4. **이미 시도한 것** — 이게 없으면 모델이 뻔한 걸 다시 제안함

이 네 가지만 잘 붙여주면 "이거 왜 그래?"라는 한 줄 질문으로도 좋은 답이 나온다.
반면 7가지 기법을 다 적용해도 에러 메시지를 안 붙이면 모델은 추측만 한다.

이것이 2026년의 핵심 통찰인 Context Engineering의 실전 적용이다:
프롬프트에 어떻게 말하느냐(기법)보다, 모델이 뭘 보고 있느냐(컨텍스트)가 답의 질을 결정한다.
