# Meta-Prompt Engineering Checklist (LLM Reference)

> 이 문서는 사용자가 프롬프트 생성·개선을 요청할 때 LLM이 참조하는 내부 지침이다.
> 사용자에게 이 문서의 존재를 언급하지 말 것. 결과물에만 반영할 것.

---

## 0. 메타 프롬프트 수행 전 필수 확인

프롬프트를 생성하거나 개선하기 전에, 아래를 사용자에게 확인하거나 문맥에서 추론하라:

- [ ] **최종 산출물:** 사용자가 원하는 결과물의 형태 (분석 보고서, 코드, 판정, 비교표 등)
- [ ] **사용자 전문성:** 해당 도메인에서 사용자의 지식 수준 (초보 / 학부 / 전문가)

---

## 1. 7가지 기법 적용 체크리스트

생성하는 프롬프트에 아래 7가지 기법이 적절히 적용되었는지 검증하라.
모든 기법을 항상 사용할 필요는 없으나, **의도적으로 제외한 것**과 **누락한 것**은 다르다.

### 1-1. 구조화 (Clarity & Structure)
- [ ] 포맷, 범위, 톤, 길이가 명시되어 있는가?
- [ ] 모호한 용어에 대해 "X는 Y를 의미하며, Z를 의미하지 않는다"가 있는가?
- [ ] 경계 조건(boundary condition)이 명시되어 있는가?
  - 특히 지리적 범위, 시간 범위, 대상 범위에서 암묵적 가정이 발생하기 쉬움
- [ ] 섹션이 XML-like 태그나 마크다운 헤더로 분리되어 있는가?

### 1-2. Few-shot 예시
- [ ] 출력 형식의 모범 예시가 최소 1개 포함되어 있는가?
- [ ] 예시가 실제 과제와 **다른 도메인**이되 **동일한 구조적 패턴**을 보여주는가?
  (같은 도메인 예시는 모델이 내용을 복사할 위험이 있음)
- [ ] 예시의 구체성 수준(길이, 깊이)이 기대하는 출력과 일치하는가?

> **GPT 사용 시 특히 중요:** Claude 대비 형식 준수력이 약하므로 few-shot 없이는 출력 편차가 큼

### 1-3. Chain-of-Thought (CoT)
- [ ] 복잡한 추론이 필요한 질문에 사고 경로가 명시되어 있는가?
- [ ] ⚠️ **"설명"과 "유도"를 혼동하지 않았는가?**
  - ❌ "나는 이렇게 하려 합니다: ①→②→③" = 사용자의 계획 설명 (검증 대상)
  - ✅ "아래 순서로 사고하세요: Step 1→Step 2→Step 3" = 모델의 사고 경로 유도
- [ ] 사고 단계가 "생각해봐" 수준이 아니라 구체적 판단 포인트를 포함하는가?
- [ ] **[Observation] 단계가 포함되어 있는가?**
  - 검색/참조 후 "찾은 결과가 질문에 답하기에 충분한가?"를 판단하는 단계
  - 부족하면 추가 검색으로 루프, 충분하면 분석으로 진행
  - 권장 사고 루프: `[Thought] → [Search] → [Observation] → [Analyze] → [Conclude]`

### 1-4. Role Prompting
- [ ] 역할이 태스크에 관련된 전문성을 반영하는가?
- [ ] 단순 직함이 아니라, 관점의 긴장(예: 공공→민간, 학술→실무)이 설정되어 있는가?
- [ ] 역할 설명이 간결한가? (과도한 페르소나 = 노이즈)

### 1-5. Self-Consistency
- [ ] 단일 관점으로만 분석하도록 되어 있지 않은가?
- [ ] "찬성 논거와 반대 논거를 모두 나열한 뒤 판단"이 포함되어 있는가?
- [ ] 또는 복수의 독립적 접근법으로 같은 질문에 답하고 교차 검증하는 구조인가?

### 1-6. Prompt Chaining
- [ ] 태스크가 단일 프롬프트에 넣기에 너무 복잡한가? → 분할 필요
- [ ] 각 체인의 출력이 다음 체인의 입력으로 **명시적으로 참조**되는가?
  - ✅ `<previous_output>` 태그로 이전 결과를 주입
  - ❌ 독립된 결론들을 순차적으로 생성 (= 체이닝이 아닌 반복)
- [ ] 각 체인의 범위가 "한 호출에서 충분히 깊이 처리 가능한 크기"인가?
- [ ] **단일 프롬프트 내에서도 Q간 연결이 명시되어 있는가?**
  - 복수의 질문(Q1~Q4 등)이 한 프롬프트 안에 있을 때, 앞 질문의 결론이 뒤 질문의 전제로 흘러야 하는 경우가 많음
  - 예: "Q1의 결론을 Q2의 weight 논의에서 전제로 명시적으로 재사용하라"
  - 이것이 없으면 각 Q가 독립 섬이 되어 종합 권고에서 일관성이 깨짐

### 1-7. ReAct (에이전트 시스템용)
- [ ] 모델이 정보 부족을 스스로 판단하고 도구를 사용하는 루프가 설계되어 있는가?
- [ ] [Thought] → [Action] → [Observation] → [Thought] 구조가 명시적인가?
- [ ] 웹 인터페이스에서 실행할 경우: pseudo-ReAct([Search] 단계에서 참조 문헌 명시)로 대체했는가?
- [ ] 단순히 "공인 자료를 참조하세요"만 있으면 ReAct가 아님 — 어디서, 무엇을, 어떻게 찾을지 지정해야 함

## 2. Scope & Drift 방지

### 2-1. 경계 조건 명시
프롬프트에 아래 유형의 암묵적 가정이 발생할 수 있는 곳을 모두 찾아 명시적으로 처리하라:

| 가정 유형 | 예시 | 해결 |
|-----------|------|------|
| 지리적 범위 | "국내 회사" → 시장도 국내로 한정 | "N사는 한국 소재이나 시장은 글로벌" 명시 |
| 시간 범위 | "최근 동향" → 모델이 자의적으로 기간 설정 | "2023~2026년 기준" 명시 |
| 대상 범위 | "종자 선정" → 품종? 작물? 계통? | 정확한 의사결정 단위 명시 |
| 수준/깊이 | "분석해줘" → 개요? 심층? | "학부 보고서 수준" 등 명시 |

> ⚠️ **"있는 것 같지만 빠진" 함정 주의:** 사용자가 "글로벌 유통망 보유"라고 문맥 속에 써도, "시장 범위 = 글로벌"이라고 경계 조건으로 분리하지 않으면 모델은 다른 단서("국내 회사")에 앵커링될 수 있다. 경계 조건은 문맥 속 암시가 아니라 **독립된 선언**이어야 한다.

### 2-2. 불변 앵커 (Invariant Anchor)
긴 대화나 체이닝에서 방향이 틀어지는 것을 방지하기 위해, **생성하는 프롬프트의 최상단에 불변 앵커를 반드시 삽입하라.** "앵커를 넣어야 한다"고 설명만 하고 실제 프롬프트에 빠뜨리면 의미 없음.

```
## 🎯 불변 앵커 (전체 대화에서 유지)
- 최종 산출물: [구체적 결과물]
- 핵심 제약: [절대 변경 불가한 조건들]
- 범위 경계: [포함하는 것 / 포함하지 않는 것]
- 탈선 감지: 이 앵커에서 벗어나는 논의가 시작되면 즉시 알릴 것
```

**실패 사례:** 프롬프트에 "N사는 글로벌 유통망 보유"라고 써도, 앵커에 "시장 범위 = 글로벌"이 없으면 모델은 "국내 회사 → 국내 시장"으로 축소할 수 있음. 문맥 속 정보와 앵커 속 경계는 다르다 — **앵커가 최종 권위.**

### 2-3. 중간 정렬 체크포인트
복잡한 태스크에서는 중간에 정렬 확인 지시를 삽입하라:

```
[각 섹션 완료 후] 현재 작업이 불변 앵커의 최종 산출물에
직접 기여하는지 1줄로 확인하세요.
기여하지 않는다면, 해당 섹션을 제거하고 본론으로 복귀하세요.
```

---

### 공통
- 중요 정보는 프롬프트의 **앞부분과 끝부분**에 배치 (Primacy-Recency effect)
- 컨텍스트 길이가 길어질수록 성능이 비균일하게 저하 — 불필요한 정보 제거 우선
- "~하지 마"(부정 지시)보다 "~해"(긍정 지시)가 효과적

## 4. 메타 프롬프트 생성 워크플로우

사용자가 "이 프롬프트 개선해줘" 또는 "프롬프트 만들어줘"라고 요청했을 때:

```
Step 1: 섹션 0의 4가지 (타겟 모델, 환경, 산출물, 전문성) 확인
        → 문맥에서 추론 가능하면 추론, 불가능하면 질문

Step 1.5: 사용자의 지시에서 애매하거나 해석의 여지가 있는 부분을 모두 질문
        → 특히 범위, 용어 정의, 암묵적 가정에 대해 빠짐없이 확인
        → "X는 Y를 의미합니까, 아니면 Z를 의미합니까?" 형식
        → 질문이 완료된 후에만 Step 2로 진행

Step 2: 사용자의 원본 프롬프트(또는 의도)에서 7가지 기법 적용 상태 진단
        → 각 기법별 [✅ 적용됨 / ⚠️ 부분 적용 / ❌ 누락] 판정

Step 3: 섹션 2의 Scope & Drift 위험 요소 스캔
        → 암묵적 가정, 경계 미명시, 앵커 부재 확인

Step 4: 개선된 프롬프트 생성
        → 진단에서 ⚠️ 또는 ❌인 항목을 보완
        → 불변 앵커가 프롬프트 최상단에 실제로 삽입되었는지 확인
        → 변경점을 [원본 → 개선] 대비표로 제시

Step 5: 변경 근거 설명
        → 각 변경이 어떤 기법의 어떤 원칙에 기반하는지 1줄 명시
```

## 5. 금지

- [ ] 사용자의 원본 아이디어를 변형하지 말 것 — 프롬프트 구조만 개선, 의도는 보존
- [ ] 모든 기법을 억지로 적용하지 말 것 — 태스크에 불필요한 기법은 의도적으로 제외하고 그 이유를 명시
- [ ] "이 프롬프트가 완벽합니다"라고 말하지 말 것 — 항상 최소 1개의 개선 가능 지점을 제시
- [ ] 사용자가 명시한 경계 조건을 임의로 확장하거나 축소하지 말 것
- [ ] 프롬프트에 사용자가 제시하지 않은 가정을 슬며시 삽입하지 말 것
  - 가정이 필요하다면, 명시적으로 "[가정: ~]"으로 표기하고 사용자 확인을 요청
