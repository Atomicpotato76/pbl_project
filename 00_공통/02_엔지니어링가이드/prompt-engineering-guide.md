## 1. 7가지 핵심 프롬프트 기법

### 1-1. Zero-shot + 구조화 (Clarity & Structure)

**정의:** 예시 없이, 지시만으로 모델을 제어하는 방식. 단 "잘 써줘" 수준이 아니라 포맷·범위·조건·맥락을 명시적으로 구조화해야 유효하다.

**핵심 원칙:**
- 원하는 **포맷, 범위, 톤, 길이**를 반드시 명시
- 2026년 최신 모델(Claude Opus 4.7 등)은 프롬프트를 더 literal하게 해석 — 말한 것만 한다
- 지시를 넓게 적용하려면 범위를 명시: "이 포맷을 모든 섹션에 적용해"

**종자회사 예시:**
```
당신은 종자회사의 R&D 기획팀 소속입니다.

향후 5년(2026–2031) 단위 프로젝트에서 신규 종자를 선정할 때 사용할
판단기준 프레임워크를 설계하세요.

조건:
- 판단기준은 최소 5개 카테고리로 구성
- 각 기준에 가중치(%) 부여
- 한국 농업 시장 맥락 반영
- 출력 형식: 마크다운 테이블 + 각 기준별 2줄 설명
```

**포인트:** "판단기준 만들어줘"와 위 프롬프트의 차이는 **모호함의 제거**에 있다. Zero-shot이라도 구조화가 충분하면 few-shot 못지않은 결과가 나온다.

---

### 1-2. Few-shot (Multishot Examples)

**정의:** 모델에게 입출력 예시를 2~5개 제공하여 출력 패턴을 문맥 내에서 학습시키는 방식.

**핵심 원칙:**
- 예시는 `<example>` 태그로 분리
- 실제 사용할 도메인과 가까운 예시가 좋지만, 다른 도메인의 **구조적 패턴**도 전이 가능
- 특히 **GPT에서는 few-shot 유무가 출력 형식 일관성에 결정적** — Claude보다 형식 준수력이 약함
- 분류, 추출, 구조화된 출력 태스크에서 거의 필수

**종자회사 예시:**
```
종자 선정 기준을 평가 매트릭스로 만들어주세요.
아래 예시 형식을 따라주세요.

<example>
입력: 의약품 원료작물 선정 기준
출력:
| 기준 | 가중치 | 평가지표 | 데이터 소스 |
|------|--------|----------|------------|
| 유효성분 함량 안정성 | 30% | 연간 성분 변동계수(CV) ≤5% | 3년간 포장시험 |
| 규제 적합성 | 25% | GMP 원료 등록 가능 여부 | 식약처 고시 |
| 재배 용이성 | 20% | 단위면적당 노동시간 | 농촌진흥청 DB |
| 수급 안정성 | 15% | 국내 자급률 + 수입 대체 가능성 | KREI 통계 |
| 경제성 | 10% | 3년 ROI ≥ 150% | 자체 시뮬레이션 |
</example>

이제 아래 과제에 같은 형식을 적용하세요:
입력: 종자회사 5년 단위 프로젝트용 종자 선정 기준
```

**포인트:** 의약품이라는 다른 도메인의 예시를 줬지만, 모델은 "가중치 + 평가지표 + 데이터소스" 테이블 구조를 전이 학습한다. 예시의 내용이 아니라 **구조적 패턴**이 핵심.

### 1-3. Chain-of-Thought (CoT) + Extended Thinking

**정의:** 모델에게 단계별 사고 경로를 명시적으로 유도하여 추론 품질을 높이는 기법.

**3가지 수준:**
- **Basic:** "단계별로 생각해봐" (효과 제한적)
- **Guided:** 구체적 사고 단계를 지정 (실전에서 가장 유용)
- **Structured:** `<thinking>`, `<answer>` 등 태그로 사고와 출력을 분리

**권장 사고 루프 (CoT + pseudo-ReAct 통합):**
`[Thought] → [Search] → [Observation] → [Analyze] → [Conclude]`

여기서 `[Observation]`이 핵심 — "찾은 근거가 충분한가?"를 판단하는 단계. 이것이 없으면 모델이 첫 검색 결과만으로 바로 결론을 내려버리는 경향이 있다.

**핵심 원칙:**
- "모델의 사고 경로"를 설계하는 것이지, "내 계획을 설명"하는 것이 아님
- 워크플로우 ①~⑤를 제시하는 것 ≠ CoT (그건 검증 대상 제시)
- CoT = "Q1에 답할 때 아래 순서로 사고해"

**⚠️ 흔한 실수 — "설명"과 "유도"의 혼동:**

```
# ❌ 이건 CoT가 아님 — "내 계획 봐줘"
나는 이렇게 하려고 합니다:
① TPP sketch → ② 시나리오 선택 → ③ 재채점 → ④ 재계산 → ⑤ 민감도 분석
이 접근이 타당한지 검증해주세요.
```

```
# ✅ 이게 CoT — "이 경로로 사고해줘"
Q1에 답할 때 아래 순서로 사고하세요:
Step 1: AHP 원전(Saaty 1980)에서 criteria-alternative 독립성 전제 확인
Step 2: 시나리오 종속 scoring이 이 전제를 위반하는지 판단
Step 3: MCDA 문헌에서 이를 허용하는 변형이 있는지 탐색
Step 4: 결론 + 수정안
```

**종자회사 예시 (Guided CoT):**
```
종자회사의 5년 프로젝트용 종자 선정 기준을 만들어야 합니다.

다음 단계를 순서대로 밟아서 사고 과정을 보여주세요:

1단계 — 이해관계자 분석:
  이 기준을 사용할 사람은 누구인가? (육종팀, 경영진, 마케팅)
  각 이해관계자가 중시하는 가치는?

2단계 — 외부 환경 스캔:
  향후 5년간 한국 농업에 영향을 줄 메가트렌드는?

3단계 — 기준 도출:
  1~2단계를 종합해서 판단기준 카테고리를 도출하세요.

4단계 — 가중치 및 측정법:
  각 기준의 상대적 중요도와 측정 가능한 지표를 설계하세요.

5단계 — 검증:
  이 프레임워크를 팥(Vigna angularis)에 시범 적용하세요.
```

**Effort 파라미터 (API 사용 시):**

| 설정 | 용도 | 비고 |
|------|------|------|
| `max` | 극도의 추론 필요 | overthinking 위험 있음 |
| `xhigh` | 코딩·에이전트 | 대부분의 복잡 작업에 최적 |
| `high` | 분석·판단 | 일반적 추론 최소 권장 |
| `medium` | 비용 절감 | 지능 트레이드오프 |
| `low` | 단순 조회·포맷팅 | 복잡한 문제에 사용 금지 |

> claude.ai 대화에서는 모델이 effort를 자동 조절함. API 파이프라인 설계 시 참고.

### 1-4. Role Prompting (역할 부여)

**정의:** 모델에 특정 전문가 역할을 부여하여 도메인 전문성 수준과 관점을 설정하는 기법.

**핵심 원칙:**
- 단순 직함보다 **경력 경로에 내재된 관점의 긴장**을 설정하면 단면적이지 않은 분석이 나옴
- 역할은 현실적이고 태스크에 관련된 것으로 간결하게 — 과도한 페르소나는 노이즈
- 역할 + CoT를 조합하면 시너지가 큼

**종자회사 예시:**
```
당신은 농촌진흥청 산하 국립종자원에서 15년간 근무한 뒤
민간 종자회사(매출 500억 규모)의 CSO(Chief Strategy Officer)로
이직한 전문가입니다.

공공 육종 시스템의 한계와 민간 R&D의 상업적 압박을
모두 이해하는 시각에서, 향후 5년 프로젝트의 종자 선정 기준을
설계해주세요.

특히 학술적 이상과 상업적 현실 사이의 트레이드오프를
명시적으로 다뤄주세요.
```

**포인트:** "종자 전문가"보다 "공공→민간 이직자"라는 설정이 공공 vs 민간의 렌즈를 겹치게 만들어, 현실적 기준이 나온다.

### 1-5. Meta Prompting & Self-Consistency

**Meta Prompting — 프롬프트 자체를 개선시키기:**
```
나는 종자회사의 5년 프로젝트에서 종자 선정 기준을 만들려고 해.
아래가 내 초안 프롬프트야:

"종자회사에서 5년 프로젝트할 때 어떤 종자를 고를지
기준을 만들어줘."

이 프롬프트를 개선해서, 더 체계적이고 실행 가능한
판단기준이 나오도록 다시 작성해줘.
개선 시 빠진 맥락, 모호한 부분, 추가해야 할 제약조건을
지적해줘.
```

**Self-Consistency — 다중 추론 경로로 교차 검증:**
```
종자 선정 기준을 3가지 서로 다른 접근법으로 각각 독립적으로
도출해주세요:

접근법 A: 시장 수요 기반 (top-down, 소비자→역추적)
접근법 B: 육종 역량 기반 (bottom-up, 보유 자원→가능성)
접근법 C: 리스크 최소화 기반 (방어적, 실패 확률→회피)

3가지 결과를 비교해서 공통으로 등장하는 기준을 핵심 기준으로,
하나의 접근법에서만 나오는 기준을 선택적 기준으로 분류해주세요.
```

**핵심:** 삼각측량(triangulation). 단일 추론 경로의 blind spot을 다른 관점이 잡아낸다.

### 1-6. Prompt Chaining (프롬프트 체이닝)

**정의:** 복잡한 태스크를 작은 단계들로 분할하여 순차적으로 연결하는 방식. 각 단계의 출력이 다음 단계의 입력이 된다.

**핵심 원칙:**
- 한 프롬프트에 모든 걸 넣으면 중간 단계를 건너뛰거나 얕게 처리할 수 있음
- 각 체인의 출력이 다음 체인의 **입력으로 명시적으로 참조**되어야 함
- API에서는 별도 호출, claude.ai에서는 "답변 받은 후 다음 단계 요청"

**⚠️ 흔한 실수 — "연속 세션"과 "체이닝"의 혼동:**

```
# ❌ 이건 체이닝이 아님 — 독립된 결론들의 나열
세션 1: "트렌드 분석해줘" → 결과 md 저장
세션 2: "판단기준 만들어줘" → 결과 md 저장
(각 결과가 서로를 참조하지 않음)
```

```
# ✅ 이게 체이닝 — 이전 출력이 다음 입력의 전제
[Chain 2 프롬프트 서두]
아래는 이전 단계(트렌드 분석)의 결과입니다:
<previous_output>
{Chain 1 결과 붙여넣기}
</previous_output>

이 트렌드 분석에서 도출된 핵심 동인 3가지를 기반으로,
종자 선정 판단기준 카테고리를 도출하세요.
각 기준이 어떤 트렌드에서 파생됐는지 매핑하세요.
```

**종자회사 예시 (4-Step Chain):**
```
[Step 1] 한국 종자산업 2026–2031 메가트렌드 분석
         (기후변화, 정책, 소비, 기술 4축)
         → <trends>에 저장

[Step 2] <trends> 기반 판단기준 카테고리 도출
         + 트렌드-기준 매핑
         → <criteria>에 저장

[Step 3] <criteria> 각 기준에 대해:
         KPI 설계 + 데이터 수집법 + 5점 rubric

[Step 4] 완성 프레임워크를 팥에 시범 적용
         → 맹점 피드백
```

### 1-7. ReAct (Reasoning + Acting + Observing)

**정의:** 모델이 "사고 → 도구 사용 → 결과 관찰 → 재사고"의 자율 루프를 반복하는 구조. 에이전트 시스템의 핵심 패턴.

**핵심 원칙:**
- Thought: 어떤 정보가 부족한지 판단
- Action: 도구(검색, DB 조회, API 호출)로 정보 수집
- Observation: 수집 결과 요약 및 충분성 판단
- 정보가 충분할 때까지 루프 반복

**종자회사 예시:**
```
당신은 종자회사 R&D 전략 에이전트입니다.

과제: 5년 프로젝트용 종자 선정 기준 수립

다음 루프를 따르세요:

[Thought] 현재 어떤 정보가 부족한지 판단
[Action] 필요한 정보를 도구로 검색
  - web_search: 시장 데이터, 정책 동향
  - 논문 검색: 육종 기술 트렌드
  - 통계 DB: FAO, KREI, 농림부 자료
[Observation] 검색 결과를 요약
[Thought] 수집된 정보를 바탕으로 기준 초안 수정
→ 정보가 충분할 때까지 반복

최종 출력: 근거 기반의 종자 선정 프레임워크
(각 기준 옆에 출처 명시)
```

**실무 제약:** GPT/Claude 웹 인터페이스에서 완전한 ReAct 루프는 어려움. 대안:
- **Pseudo-ReAct:** `[Search]` 단계에서 "어떤 문헌을 참조했는지 명시"하게 하여 근거 추적성 확보
- **진짜 ReAct:** Claude API + tool use, 또는 LangChain/LlamaIndex 기반 에이전트 파이프라인

## 2. 기법별 용도 비교

| 기법 | 최적 상황 | 난이도 | 단독 사용 | 조합 추천 |
|------|----------|--------|----------|----------|
| Zero-shot + 구조화 | 단순 질문, 잘 정의된 태스크 | 낮음 | ✅ | 모든 기법의 기반 |
| Few-shot | 분류, 포맷 통제, 추출 | 중간 | ✅ | + CoT, + Role |
| CoT | 수학, 논리, 다단계 분석 | 중간 | ✅ | + Role, + Few-shot |
| Role | 도메인 전문성 조절 | 낮음 | ⚠️ 단독 약함 | + CoT, + 구조화 |
| Meta / Self-consistency | 프롬프트 개선, 품질 검증 | 중간 | ✅ | + Few-shot |
| Chaining | 복잡한 워크플로우 분할 | 높음 | ✅ | + CoT, + Role |
| ReAct | 에이전트, 도구 사용 시스템 | 최상 | ⚠️ API 필요 | + Chaining |

**실전 조합 예시 (PBL 프로젝트):**
> Role(종자원 출신 CSO) + CoT(5단계 사고) + Few-shot(평가 매트릭스 예시) + Chaining(트렌드→기준→스코어카드→시범적용)

## 3. 실전 프롬프트 진단 — MCDA 종자 선정 프롬프트

### 3-1. 원본 프롬프트 진단

| 기법 | 적용 상태 | 판정 | 문제점 |
|------|----------|------|--------|
| Zero-shot + 구조화 | Q1~Q4 명확, 출력형식 지정 | ✅ 강함 | — |
| Few-shot | 없음 | ⚠️ 부재 | 판정 형식의 예시 없어서 모델마다 출력 편차 |
| CoT | 워크플로우 ①~⑤ 제시 | ⚠️ 착각 | "내 계획 설명"이지 "모델 사고 경로 유도"가 아님 |
| Role | MCDA 전문가 + 종자산업 컨설턴트 | ✅ 강함 | — |
| Meta | 다른 세션에서 프롬프트 생성 | ✅ 적용 | — |
| Self-consistency | 없음 | ⚠️ 부재 | 찬반 논거 교차검증 없음 |
| Chaining | md 저장 → 수동 붙여넣기 | ⚠️ 비효율 | 누적 구조 아닌 직렬 반복 |
| ReAct | 미적용 | ❌ 부재 | 검색 지시·정보 판단·관찰 루프 전무 |

### 3-2. 핵심 문제 3가지

**문제 1 — CoT가 "설명"이지 "유도"가 아님**

워크플로우 ①~⑤는 "내가 이렇게 하려는데 맞냐?"라는 검증 대상으로 제시된 것이다. 모델은 이걸 "평가해달라는 것"으로 읽지, "따라야 할 사고 순서"로 읽지 않는다.

CoT라면 "Q1에 답할 때 아래 순서로 사고해" + 구체적 사고 경로를 지정해야 한다.

**문제 2 — Few-shot 부재로 출력 품질 도박**

`[판정: ✅/⚠️/❌] + [근거] + [수정 제안]` 형식을 지정했지만 모범 답안 예시가 없어서:
- 근거의 길이가 1줄~10줄 사이에서 변동
- "수정 제안"의 구체성 수준이 불일치
- 판정 기준(뭘 ✅로, 뭘 ⚠️로 보는지)이 자의적

**문제 3 — ReAct 부재로 레퍼런스가 장식**

참고문헌 4개를 직접 제시했지만, 모델이 이걸 실제로 찾아서 읽고 판단하는 구조가 아님. Thought-Action-Observation 루프 없이 "레퍼런스 쓰세요"만 있으면 모델은 "이런 논문이 있다고 했구나" 수준으로만 활용.

## 6. LLM과 협업할 때 기억할 것

### LLM이 잘하는 것
- 제시된 구조 안에서 밀도 높은 분석
- 명시적으로 요청된 관점 간 비교
- 대량의 정보를 일관된 형식으로 정리
- 사용자가 제시한 프레임워크의 내적 일관성 검증

### LLM이 못하는 것
- **비요청 연결 (unsolicited association):** 사용자가 두 개념을 같은 프롬프트에 넣어줘야 비로소 관련성을 인식. "샤워하다 갑자기 떠오르는" 종류의 창의적 연결 불가능.
- **능동적 지식 탐색:** "뭘 모르는지"를 스스로 판단하고 찾아가는 능력이 약함. ReAct 루프를 명시적으로 설계해줘야 비로소 작동.
- **메타 프롬프트의 base case 생성:** "프롬프트를 만들기 위한 프롬프트를 만들기 위한..." 재귀의 종점은 항상 사용자의 원초적 아이디어. LLM은 렌즈를 고를 수 있지만, 어디를 찍을지는 사람이 결정.

### 프롬프트 엔지니어링의 본질
프롬프트 엔지니어링은 사용자 머릿속에서 이미 일어난 연결을 LLM이 이해할 수 있는 구조로 번역하는 작업이다. 기법은 번역의 정밀도를 높여주지만, 번역할 원본(아이디어)은 사용자에게서 나온다.

## 7. 2026년 패러다임 전환: Context Engineering

### Prompt Engineering vs Context Engineering

| 구분 | Prompt Engineering | Context Engineering |
|------|-------------------|---------------------|
| 초점 | 모델에게 **어떻게 말하느냐** | 모델이 **뭘 보고 있느냐** |
| 비유 | 요리사에게 주문하는 기술 | 냉장고에 뭘 채우느냐 |
| 해결하는 문제 | 단일 응답의 품질 | 시스템 수준의 신뢰성 |
| 주요 기술 | 위 7가지 기법 | RAG, 메모리 관리, 토큰 배분, 도구 오케스트레이션 |
| 2026년 위치 | table stakes (기본기) | 차별화 요소 |

### 핵심 인사이트
- 2026년 프론티어 모델들의 instruction following이 거의 완성 단계에 도달하면서, 프롬프트 자체는 "싸졌고" 고품질 컨텍스트가 "비싸졌다"
- Gartner(2025): "context engineering is in, prompt engineering is out"
- 그러나 프롬프트 엔지니어링이 불필요해진 게 아님 — context engineering은 prompt engineering을 포함하면서 확장한 것
- Chroma Research(2025.07): 18개 LLM 테스트 결과, 모델은 컨텍스트를 균일하게 사용하지 않고 입력 길이 증가에 따라 성능이 비균일하게 저하
- **실전 팁:** 긴 프롬프트에서 중요한 정보는 앞과 끝에 배치

| Q간 연결 없음 | ✅ 반영 완료 | "앞 Q 결론을 뒤 Q 전제로 재사용" 명시 |

### 실험에서 얻은 교훈
- 체크리스트를 LLM에게 평가 기준으로 제공하면, 인간이 직관적으로 느끼는 약점을 체계적으로 짚어낼 수 있음
- 그러나 LLM은 **긍정 편향(sycophancy)**이 있어 약간 후하게 채점하는 경향 — 인간의 보정 필수
- "있는 것 같지만 빠진" 경계조건은 LLM도 놓칠 수 있음 (GPT가 경계조건에 4점을 줬지만 실제로는 scope drift 발생)
