# 후속 로드맵 실무 유사성 점검

문서 상태: COMMON
질문: 후속 로드맵이 실제 논문이나 실무와 비슷한 과정으로 진행되었는가?

## 결론

현재 후속 로드맵의 큰 흐름은 실제 표준유전체 프로젝트와 대체로 유사하다. 특히 `기존 유전체 현황 점검 → 기준 계통 선정 → long-read 기반 조립 → Hi-C scaffolding → RNA-seq 기반 주석 → QC → 변이 분석 → 활용 연결` 순서는 실제 논문과 대형 genome project 표준과 맞는다.

다만 발표용으로는 과도하게 세부적인 수치 비교나 기술사 설명을 줄여야 한다. 실무와 비슷하게 보이려면 기술명을 많이 나열하는 것보다, 각 데이터가 왜 필요한지 역할을 분리해 설명해야 한다.

## 실제 과정과 맞는 부분

- `Genome status audit`: 기존 reference, draft, pangenome, 품질 지표를 먼저 확인하는 것은 타당하다.
- `Reference strategy`: de novo, upgrade, resequencing, pangenome 중 하나를 고르는 구조는 타당하다.
- `Sequencing design`: long-read와 Hi-C, RNA-seq 조합은 현대적 reference genome 설계와 맞다.
- `Assembly/QC/Annotation`: 조립, scaffolding, annotation, BUSCO/N50/read mapping 점검은 실제 논문 흐름과 유사하다.
- `Use-case bridge`: breeding, seed pathway, procurement/adoption으로 연결하는 것은 N사 프로젝트 맥락에 필요하다.

## 보완할 부분

- 블록 1에서 작물의 재배 문제와 공급 루트를 먼저 닫고, 블록 2에서 그 문제를 분석 질문으로 바꿔야 한다.
- `pangenome`, `T2T`, `AI`는 필요할 때만 확장 카드로 넣고 발표 본론으로 키우지 않는다.
- `N50`, `BUSCO`, `LAI` 같은 지표는 품질 확인 예시로만 쓰고 수치 경쟁을 하지 않는다.
- 실제 실무처럼 보이려면 `어떤 샘플을 왜 읽는가`, `어떤 형질을 왜 분석하는가`, `결과를 어디에 쓰는가`가 먼저 보여야 한다.
