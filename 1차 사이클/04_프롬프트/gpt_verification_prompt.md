## 프롬프트 시작

당신은 연구 방법론 및 프롬프트 엔지니어링 전문가입니다. 아래에 첨부된 "Research Verification Framework v1.0"을 평가해주세요.

### 배경

이 프레임워크는 대학 PBL(Problem-Based Learning) 프로젝트에서 사용됩니다. 구체적 상황:

- 종자회사의 표준유전체 해독 대상 작물을 선정하는 프로젝트
- AI(GPT Pro + Claude) 하네스를 써서 데이터 수집 → 점수 매기기 → 최종 선정을 진행
- 문제 발견: GPT Pro가 생산한 데이터(시장 규모, CAGR, 정책 동향 등)를 원문 검증 없이 의사결정에 사용함
- "주장→근거→결론"에서 근거 레이어가 비어있는 상태를 구조적으로 해결하려는 목적

### 평가 요청 (아래 7개 항목 각각에 대해 구체적으로 답변)

**1. 구조적 완전성**
- 5대 축(Traceability, Independent Verification, Chain Integrity, Completeness≠Correctness, Falsifiability)이 조사 검증에 필요한 축을 충분히 커버하는가?
- 빠진 축이 있다면 무엇인가? (예: 시간적 유효성/temporal validity, 맥락 적합성/contextual fit 등)

**2. 각 축의 논리적 건강성**
- 각 축의 "원칙 → 적용 규칙 → 검증 질문" 체인이 논리적으로 일관되는가?
- 적용 규칙이 원칙을 실제로 구현하는가, 아니면 원칙과 규칙 사이에 비약이 있는가?

**3. 출처 신뢰등급 체계**
- 🟢/🟡/🔴 3단계 분류가 이 프로젝트 맥락에서 적절한가?
- 등급 간 경계가 모호한 케이스가 있는가? (예: FAOSTAT의 추정치 vs 실측치)
- 개선 제안이 있다면?

**4. 실행 가능성**
- "Stage 0 추가 지시", "Stage 2 검증 절차", "Gate Check 추가 조건"이 실제로 실행 가능한가?
- 학부 2학년이 운영하는 AI 하네스에서 과부하(overkill)가 되는 부분이 있는가?
- 시간 대비 효과가 낮은 규칙이 있다면 지적해주세요.

**5. 태그 체계**
- 기존 태그([실측], [시장추정], [전문가추론], [미확보], [문헌확인])에 새 태그([미검증], [체인 미검증], [반례 발견])를 추가했는데, 태그가 너무 많아서 실용성이 떨어지지 않는가?
- 태그 간 중복이나 혼동 가능성이 있는가?

**6. 프롬프트 엔지니어링 관점**
- 이 프레임워크를 AI 하네스의 Stage 0 명령서에 통합할 때, 프롬프트로서의 효과를 높이려면 어떻게 구조화해야 하는가?
- 불변 앵커("교수님이 레퍼런스는? 하면 답할 수 있는가?")가 효과적인가?
- Primacy-Recency 관점에서 정보 배치에 문제가 있는가?

**7. 이 프레임워크 자체의 반증 가능성**
- 이 프레임워크가 "오히려 해가 되는" 시나리오가 있다면 무엇인가?
- 검증 과부하로 본 작업(작물 선정)이 지연되는 리스크를 어떻게 관리해야 하는가?
- 이 프레임워크를 적용하지 않는 것이 더 나은 상황이 있는가?

### 출력 형식

각 항목(1~7)에 대해:
- **판정**: PASS / CONDITIONAL PASS / FAIL
- **근거**: 2-3문장
- **개선 제안**: 있으면 구체적으로 (없으면 "없음")

마지막에 **종합 평가**를 추가해주세요:
- 전체 PASS/FAIL 판정
- 가장 시급한 개선 3가지 (우선순위 순)
- "이 프레임워크를 한 문장으로 요약하면?"
