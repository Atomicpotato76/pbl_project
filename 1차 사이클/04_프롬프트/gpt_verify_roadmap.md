### 축 8 세부: 프롬프트 엔지니어링 원칙 체크

Stage 0 명령서 각각에 대해 아래 10개 원칙의 준수 여부를 확인하라:

| # | 원칙 | 확인 방법 |
|---|------|----------|
| P1 | Tier 판단 | 명령서의 복잡도에 맞는 기법이 적용되었는가? 과잉 프롬프팅이 없는가? |
| P2 | 과잉 프롬프팅 금지 | 불필요하게 복잡한 구조나 과도한 제약이 있는가? |
| P3 | 설명 vs 유도 구분 | 명령서가 "결과를 설명하라"와 "이 순서로 사고하라"를 적절히 구분하는가? |
| P4 | Observation 단계 | 조사 후 "이 답이 충분한가?" 자기 검증 지시가 있는가? |
| P5 | 불변 앵커 | 각 명령서가 PBL 문제와의 연결을 유지하는가? scope drift 방지가 있는가? |
| P6 | Few-shot 도메인 분리 | few-shot 예시가 같은 도메인 내용 복사가 아닌 "검증 과정"을 보여주는가? |
| P7 | Chaining 명시 참조 | MC 간 결과 참조가 "[MC X-Y 결과 참조]"로 명시되어 있는가? |
| P8 | Context > Prompt | seed 자료(기존 문서)를 프롬프트 지시보다 맥락으로 우선 제공하는가? |
| P9 | Primacy-Recency | 핵심 지시가 명령서의 앞/끝에 배치되어 있는가? |
| P10 | 긍정 지시 | "~하지 마"보다 "~하라"가 우세한가? |

---

각 축에 대해:
1. PASS / FAIL 판정
2. 구체적 근거 (어떤 MC의 어떤 부분)
3. FAIL 시 수정 방향 1-2줄

종합 판정:
- **PASS**: 8개 축 모두 PASS
- **CONDITIONAL PASS**: FAIL이 있으나 경미 → 실행 가능, 수정 권장
- **FAIL**: 핵심 축(2, 3, 4)에서 FAIL → 수정 후 재검증
